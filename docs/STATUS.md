# Статус проекта РА «Лидер»

Дата обновления: 2026-06-23.

## Основной контур

- основной репозиторий: `deputat36/lider-bsk`;
- временная рабочая CRM v4: `https://deputat36.github.io/lidercalculator/app-v4.html`;
- основной CRM-контур: `https://deputat36.github.io/lider-bsk/crm/v4/`;
- публичный сайт: `https://www.lider-bsk.ru`;
- Supabase project: `ofewxuqfjhamgerwzull`.

Для задач РА «Лидер» использовать только объекты `leader_*`. Объекты `nav_*` относятся к другому проектному контуру.

## Supabase

Активные функции:

- `leader-public-lead v7`, публичная, `verify_jwt=false`;
- `leader-crm-leads v8`, `verify_jwt=true`;
- `leader-crm-orders v2`, `verify_jwt=true`.

Выполнено:

- CRM использует Edge Functions с JWT вместо прямого вызова закрытых RPC;
- создание заказа из согласованного КП идёт через `leader-crm-leads` action `create_order_from_offer`;
- список и изменение заказов идут через `leader-crm-orders`;
- прямой доступ `anon` и `authenticated` к служебным RPC закрыт;
- RLS-helper функции перенесены в приватную схему `leader_private`;
- права таблиц заявок, аудита и событий приведены к минимальной модели;
- таблицы событий заявки и КП работают как append-only для клиентского контура;
- Security Advisor не показывает предупреждений по `leader_*`.

Оставшиеся предупреждения относятся к `nav_*` и настройке Auth leaked password protection. Они не изменялись.

## Авторизация CRM

В основной и временной CRM:

- обрабатывается `refresh_token_not_found`;
- устаревшая локальная сессия автоматически очищается;
- выход выполняется с `scope: 'local'` и не завершает сессии на других устройствах;
- сбой сети при выходе не оставляет интерфейс в состоянии активной сессии и не создаёт необработанное исключение;
- `auth.js` подключён с cache-buster `v=20260623-session-2`;
- добавлен workflow `CRM auth checks`.

## Публичные заявки и аудит

`leader-public-lead v7`:

1. принимает отправку сайта;
2. генерирует или принимает `request_id`;
3. нормализует телефон;
4. проверяет honeypot и обязательные поля;
5. записывает заявку с защитой от дублей по `request_id`;
6. записывает событие в `leader_public_lead_audit`;
7. проверяет HTTP-результат записи аудита;
8. журналирует неблокирующие ошибки аудита маркерами `leader_public_lead_audit_insert_failed` и `leader_public_lead_audit_request_failed`.

Ошибка аудита не блокирует основную заявку.

Проверено:

- RLS-политика вставки аудита ограничивает форму публичного события;
- `anon` имеет только `INSERT`;
- `authenticated` имеет только `SELECT`;
- rollback-smoke-test вставки под `anon` прошёл без постоянной тестовой записи;
- последняя существующая заявка была создана до развёртывания функции с аудитом;
- таблица аудита пока пуста, поэтому требуется одна реальная контрольная отправка формы.

## Ограничения

- временную CRM в `lidercalculator` не отключать до завершения браузерной проверки;
- старую CRM v2 не удалять без отдельного подтверждения;
- `nav_*` не использовать;
- данные, таблицы, политики и функции не удалять без подтверждения владельца;
- POST-проверки GitHub Pages и Edge Function из текущего окружения блокируются внешним фильтром, поэтому реальная отправка формы должна быть выполнена в обычном браузере.
