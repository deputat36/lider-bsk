## Что уже применено в Supabase

### Pending-доступ новых пользователей

В базе создана модель приглашений:

- `leader_user_invites`;
- trigger активации профиля по приглашению;
- обновлённая логика `leader_ensure_profile(text)`;
- новые пользователи без приглашения получают неактивный профиль и ждут подтверждения владельцем или администратором.

Контрольная проверка после применения:

- активных профилей: 4;
- pending-профилей: 0;
- активных owner/admin: 3;
- приглашений: 0;
- политика вставки приглашений есть;
- trigger приглашений установлен.

### Атомарное создание заказа из КП

В базе создана RPC:

`leader_create_order_from_offer_rpc(jsonb)`

Назначение:

- принять `offer_id` согласованного КП;
- заблокировать КП, расчёт и заявку на время операции;
- проверить статус `Согласовано`;
- вернуть уже созданный заказ, если он уже связан с КП/расчётом/заявкой;
- создать или найти клиента;
- создать заказ;
- перенести позиции расчёта в позиции заказа;
- связать КП, расчёт и заявку с заказом;
- записать событие КП и историю статуса заказа.

Edge Function `leader-crm-leads` уже задеплоена в Supabase как version 9, `verify_jwt=true`, `ACTIVE`. Действие `create_order_from_offer` в боевом Supabase-контуре вызывает эту RPC.

## Почему есть расхождение

Supabase-операции были успешно применены через Supabase migration/deploy. Но GitHub connector несколько раз заблокировал сохранение больших SQL/Edge файлов. Поэтому база и Edge Function уже обновлены, а полная фиксация всех файлов в актуальной GitHub-ветке требует отдельного безопасного переноса.

## Что сделать перед финальным merge

1. Не мержить вслепую старый PR #11, пока не проверен конфликт с актуальным `main`.
2. Либо подтянуть `main` в `crm-hardening-20260626`, либо перенести все изменения в `crm-hardening-main-20260626`.
3. Добавить SQL миграции вручную или через локальный Git:
   - `leader_crm_pending_user_invites`;
   - `leader_create_order_from_offer_rpc`.
4. Убедиться, что `supabase/functions/leader-crm-leads/index.ts` в GitHub соответствует deployed version 9.
5. Проверить вручную:
   - вход активного пользователя;
   - pending-доступ нового пользователя;
   - вкладку `Доступ`;
   - создание и закрытие приглашения;
   - сценарий заявка → расчёт → КП `Согласовано` → заказ;
   - повторное создание заказа из того же КП без дубля.

## Безопасное текущее состояние

Боевой Supabase-контур уже защищает новых пользователей и атомарно создаёт заказ из согласованного КП. Основной риск сейчас не в базе, а в несинхронизированном GitHub-репозитории: часть применённых SQL-изменений пока зафиксирована только в Supabase и в описании PR.
