# Усиление базы CRM «Лидер» после аудита

Дата: 2026-06-21.

Этот документ описывает безопасный следующий шаг для публичных заявок с сайта. Это не применённая миграция, а техническое задание для миграции Supabase после проверки актуальной схемы таблицы `leader_leads`.

## Цель

Сделать так, чтобы одна отправка формы с сайта не могла случайно создать несколько одинаковых заявок в CRM.

## Что добавить в `leader_leads`

- `request_id` — уникальный идентификатор отправки формы.
- `phone_normalized` — телефон в нормализованном виде для поиска дублей.
- `source_page_url` — полный адрес страницы, с которой пришла заявка.
- `source_page_path` — путь страницы без домена.
- `submitted_at` — время отправки формы на стороне сайта.
- `client_user_agent` — техническая информация браузера.

## Какие индексы нужны

1. Уникальный индекс по `request_id`, если поле заполнено.
2. Индекс по `phone_normalized` и дате создания для поиска свежих дублей.

## Что должна делать Edge Function `leader-public-lead`

1. Проверять honeypot-поле.
2. Проверять наличие `request_id`.
3. Нормализовать телефон.
4. Проверять повторную заявку по `request_id`.
5. Возвращать безопасный ответ при дубле, не создавая новую заявку.
6. Записывать отказанные или подозрительные отправки отдельно, если будет добавлена таблица аудита.

## Ограничения

- Не трогать `nav_*`.
- Не применять изменения к боевой базе без проверки текущей структуры.
- Не удалять существующие поля и политики.
- Не менять Supabase Auth, потому что он используется несколькими проектами.
