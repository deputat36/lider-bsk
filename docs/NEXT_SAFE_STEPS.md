# Следующие безопасные шаги по РА «Лидер»

Дата: 2026-06-24.

## Текущее состояние

- публичный сайт: `https://www.lider-bsk.ru`;
- временная CRM v4: `https://deputat36.github.io/lidercalculator/app-v4.html`;
- основной CRM-контур: `https://deputat36.github.io/lider-bsk/crm/v4/`;
- основной репозиторий: `deputat36/lider-bsk`;
- временный репозиторий CRM: `deputat36/lidercalculator`;
- Supabase project: `ofewxuqfjhamgerwzull`.

Использовать только таблицы, функции и политики `leader_*`. Объекты `nav_*` относятся к другому проектному контуру.

## Что уже сделано

1. Публичные заявки:
   - добавлены `request_id`, источник и технические поля;
   - защита от дублей работает через `on_conflict=request_id`;
   - повторный клик блокируется на клиенте;
   - `leader-public-lead v7` проверяет результат записи аудита и журналирует неблокирующие ошибки.

2. Аудит заявок:
   - используется `leader_public_lead_audit`;
   - права ограничены моделью `anon INSERT`, `authenticated SELECT`;
   - rollback-smoke-test под `anon` прошёл;
   - диагностические маркеры: `leader_public_lead_audit_insert_failed` и `leader_public_lead_audit_request_failed`;
   - CRM показывает referer и раскрываемые технические данные.

3. Supabase CRM:
   - `leader-crm-leads v8` активна с JWT;
   - `leader-crm-orders v2` активна с JWT;
   - подготовка профиля и создание заказа выполняются через Edge Functions;
   - прямой клиентский доступ к служебным RPC закрыт;
   - Security Advisor больше не показывает предупреждений по `leader_*`.

4. Авторизация CRM:
   - устаревшая сессия с `refresh_token_not_found` очищается автоматически;
   - выход использует локальный scope и не завершает другие устройства;
   - резервная очистка ограничена ключом `leader_crm_v4_session`;
   - удаление всех `sb-*` и `supabase` ключей запрещено CI;
   - session-reset подключён с `v=20260624-isolated-1`;
   - во временном Supabase-клиенте refresh token обновляется через единый `refreshPromise`;
   - во временный клиент добавлен `auth.getUser()` для диагностики.

5. CRM v4:
   - перенесены заявки, карточка, история, потребности, расчёты и КП;
   - работают создание заказа, список и карточка заказа;
   - подключены контроль заказов, финансы, производство, монтаж и контроль контактов;
   - подключены аудит заявок, дашборд и самодиагностика.

6. Подготовлены документы:
   - `docs/CRM_V4_TEST_ACCESS.md`;
   - `docs/CRM_V4_TESTER_CHECKLIST.md`;
   - `docs/CRM_ADMIN_TESTER_ONBOARDING.md`;
   - `docs/CRM_V4_BROWSER_TEST_REPORT.md`;
   - `docs/PUBLIC_LEAD_AUDIT.md`.

## Первая ручная проверка

1. Открыть временную CRM:

`https://deputat36.github.io/lidercalculator/app-v4.html`

2. Нажать Ctrl + F5.
3. Выполнить вход → выход → повторный вход.
4. Нажать `Проверить CRM`.
5. Убедиться, что `Supabase Auth`, профиль и чтение таблиц показывают `OK`.
6. Проверить, что выход из CRM не завершил сессии других приложений на `deputat36.github.io`.
7. Повторить вход и выход в основном CRM-контуре.
8. Проверить отсутствие 404 по `assets/v4/*.js` и `assets/v4/*.css`.

## Проверка публичной заявки и аудита

Таблица аудита пока не содержит подтверждённой реальной отправки после внедрения `v7`.

1. Открыть `https://www.lider-bsk.ru`.
2. Отправить заявку с пометкой `Тест CRM v4 audit v7`.
3. Открыть CRM и раздел `Аудит заявок`.
4. Нажать `Обновить аудит`.
5. Сопоставить заявку и событие по `request_id`.
6. Проверить телефон, страницу, UTM, user-agent, referer и `Технические данные`.
7. Если события нет, проверить маркеры ошибок Edge Function.
8. Зафиксировать результат в `docs/CRM_V4_BROWSER_TEST_REPORT.md`.

## Полный бизнес-сценарий

После проверки авторизации пройти:

`заявка → потребность → расчёт → КП → согласование → заказ → финансы → производство → монтаж`.

Проверить повторное создание заказа, связанный заказ, печать КП и производственных листов, сохранение статусов и возврат в связанную заявку.

## Ограничения

- временную CRM не отключать до завершения проверки;
- основной адрес CRM не считать окончательно переключённым до полного сценария;
- старую CRM v2 не удалять без подтверждения;
- `nav_*` не использовать;
- live Supabase не менять без понятного плана, миграции и проверки;
- POST-проверки из текущего окружения блокируются внешним фильтром;
- результат push-запусков GitHub Actions нужно проверять в интерфейсе Actions, поскольку combined status API возвращает пустой список.
