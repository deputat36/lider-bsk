# Проект РА «Лидер»

## Назначение

Проект РА «Лидер» объединяет публичный сайт, обработку заявок, внутреннюю CRM и производственный контур рекламного агентства.

Цель системы — провести клиента по цепочке:

`сайт → заявка → потребность → расчёт → коммерческое предложение → заказ → производство → монтаж → оплата → прибыль`.

## Основной репозиторий

`deputat36/lider-bsk`

Здесь должны находиться:

- публичный сайт РА «Лидер»;
- CRM «Лидер»;
- документация проекта;
- миграции Supabase;
- служебные инструкции для разработки.

## Дополнительный репозиторий

`deputat36/lidercalculator`

Используется для отдельного калькулятора и временно содержит часть CRM v4. План — перенести CRM v4 в `lider-bsk/crm/`, сохранив текущую рабочую версию до завершения проверки.

## Supabase

Используется проект Supabase с рабочими таблицами CRM «Лидер» с префиксом `leader_`.

Таблицы `nav_*` относятся к отдельному проекту «Навигатор сделок» и не должны использоваться для данных CRM «Лидер».

## Ключевые сущности CRM

- `leader_user_profiles` — профили пользователей и роли.
- `leader_role_permissions` — справочник ролей и разрешений.
- `leader_leads` — заявки.
- `leader_clients` — клиенты.
- `leader_lead_needs` — потребности клиента по заявке.
- `leader_lead_calculations` — сохранённые расчёты.
- `leader_lead_calculation_items` — позиции расчёта.
- `leader_commercial_offers` — коммерческие предложения.
- `leader_commercial_offer_events` — история событий КП.
- `leader_orders` — заказы.
- `leader_order_items` — позиции заказов.
- `leader_payments` — оплаты.
- `leader_expenses` — расходы.

## Основные пользовательские роли

- `owner` — владелец.
- `admin` — администратор.
- `manager` — менеджер.
- `designer` — дизайнер.
- `production` — производство.
- `installer` — монтажник.

Роли и права нельзя менять самостоятельно. Управление ролями должно проходить через владельца или администратора.

## Текущий технический ориентир

CRM v4 должна стать основной внутренней CRM. Старая CRM v2 используется как рабочий резерв и источник проверенной логики до завершения переноса.

## Принципы развития

- Сначала безопасность и устойчивость.
- Затем удобство ежедневной работы.
- Затем новые функции.
- Не смешивать проекты и таблицы.
- Не удалять рабочие данные без отдельного подтверждения владельца.
