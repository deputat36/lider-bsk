# Архитектурные решения

Дата обновления: 2026-06-26.

## ADR-001. Основной репозиторий проекта

Решение: основным репозиторием проекта РА «Лидер» является `deputat36/lider-bsk`.

Причина: сайт, CRM, документация и миграции должны храниться в одном месте, чтобы не путать проект с отдельным калькулятором и другими разработками.

## ADR-002. Дополнительный репозиторий калькулятора

Решение: `deputat36/lidercalculator` используется только для отдельного калькулятора.

Временное исключение: CRM v4 пока размещена там исторически. Её нужно перенести в `lider-bsk/crm/` без удаления рабочей версии до завершения проверки.

## ADR-003. Префикс таблиц CRM

Решение: все рабочие таблицы CRM «Лидер» используют префикс `leader_`.

Таблицы `nav_*` относятся к проекту «Навигатор сделок» и не используются в CRM «Лидер».

## ADR-004. Безопасность ролей

Решение: пользователь не должен самостоятельно менять свою роль, активность и набор разрешений.

Управление ролями выполняют владелец и администратор в рамках своих прав.

## ADR-005. Порядок переноса CRM v4

Решение: перенос CRM v4 выполняется поэтапно.

Порядок:

1. Скопировать CRM v4 в `lider-bsk/crm/`.
2. Проверить вход и полный рабочий сценарий.
3. Оставить старый адрес как резерв.
4. После подтверждения владельца переключить основной адрес.
5. Удаление старой версии выполнять только отдельной задачей.

## ADR-006. Приоритеты разработки

Решение: сначала устойчивость и безопасность, затем удобство, затем новые функции.

Текущий порядок:

1. Безопасность доступа.
2. Документация и структура репозитория.
3. Перенос CRM v4.
4. Мобильный интерфейс.
5. Карточка заказа.
6. Производство, монтаж, оплаты и прибыль.

## ADR-007. Улучшения после аудита сайта и CRM

Решение: улучшения после аудита внедряются сначала через безопасный контур: документация, CI-проверки, клиентская защита формы и подготовка миграций. Прямые изменения боевой базы выполняются только через миграции после проверки схемы.

Причина: сайт и CRM уже работают как единая воронка, поэтому нельзя ломать рабочий сценарий. Сначала нужно защитить индексацию, исключить случайную публикацию секретов, уменьшить риск дублей заявок и подготовить серверную идемпотентность.

Порядок:

1. Зафиксировать выводы аудита в документации.
2. Добавить автоматические проверки `robots.txt`, `sitemap.xml` и отсутствия секретов.
3. Усилить публичную форму заявок через `request_id` и защиту от повторного клика.
4. Подготовить миграцию для `leader_leads.request_id` и индексов.
5. Перенести создание заказа из согласованного КП в одну серверную атомарную операцию.
6. Завершить перенос CRM v4 в основной репозиторий.

## ADR-008. Идемпотентность публичных заявок и явный audit дублей

Решение: каждая заявка с сайта получает `request_id`. Таблица `leader_leads` хранит этот идентификатор с уникальным ограничением. Edge Function `leader-public-lead v8` не маскирует повторную отправку под новую заявку, а явно фиксирует конфликт как audit-событие `duplicate`.

Причина: пользователь может дважды нажать кнопку, браузер может повторить запрос, а мобильная сеть может дать нестабильную отправку. CRM должна получить одну заявку, но технический аудит должен показывать факт повторной отправки, чтобы не путать дубль с новой заявкой.

Реализация:

1. На сайте `assets/public-lead-form.js` генерирует `request_id` для каждой отправки.
2. Edge Function `leader-public-lead` принимает `request_id`, нормализует телефон и записывает технические данные источника.
3. В `leader_leads` добавлены поля для `request_id`, нормализованного телефона, пути страницы, времени отправки и user agent.
4. В `leader_leads` действует уникальное ограничение на `request_id`.
5. Новая заявка пишет audit-событие `accepted`.
6. Повторный `request_id` пишет audit-событие `duplicate` с причиной `request_id_conflict` и возвращает клиенту `200 OK` с тем же номером обращения.
7. JWT у публичной функции оставлен выключенным, потому что форма сайта должна принимать заявки от неавторизованных посетителей.

## ADR-009. Трассировка request_id через read-only view

Решение: для диагностики цепочки публичной заявки используется read-only view `public.leader_request_trace` и CRM-виджет `Проверить request_id` в разделе `Аудит заявок`.

Причина: тестировщику и менеджеру нужен быстрый способ проверить, где находится разрыв цепочки `сайт → leader-public-lead → leader_leads → leader_public_lead_audit → CRM`, но браузерный код не должен получать service-role ключ и не должен обходить RLS.

Реализация:

1. View `leader_request_trace` связывает `leader_leads` и последнее audit-событие из `leader_public_lead_audit` по `request_id`.
2. View создана с `security_invoker = true`, чтобы сохранялась RLS базовых таблиц.
3. `anon` и `public` не имеют прав на view.
4. `authenticated` имеет только `SELECT`.
5. CRM-виджет читает view через обычный `supabaseClient` текущего пользователя.
6. Виджет показывает статусы `complete`, `lead_without_audit`, `audit_without_lead`, `missing` в понятном виде.
7. Защита реализована через GitHub Actions проверки `Request trace view check` и `Public lead audit helper check`.

## ADR-010. Доступ новых пользователей CRM только через приглашение

Решение: новый пользователь CRM «Лидер» не получает активный доступ автоматически. При первом входе профиль создаётся в `leader_user_profiles`, но активируется только при наличии действующего приглашения в `leader_user_invites`. Без приглашения профиль остаётся неактивным и ждёт подтверждения владельцем или администратором.

Причина: прежняя логика автоматически создавала первого пользователя как `owner`, а последующих — как активных `manager`. Это удобно для старта, но рискованно для боевого CRM-контура: любой пользователь с созданным Auth-аккаунтом не должен автоматически попадать в рабочие заявки, заказы, финансы и производство.

Реализация:

1. Таблица `leader_user_invites` хранит email, роль, срок действия, кто пригласил и факт принятия приглашения.
2. Trigger `leader_apply_profile_invite_trg` на `leader_user_profiles` проверяет активное приглашение при создании профиля.
3. Если приглашение найдено, профиль получает указанную роль и `is_active = true`, а приглашение помечается использованным.
4. Если приглашения нет, профиль создаётся с ролью `manager` или `owner` для пустого проекта, но всегда с `is_active = false`.
5. Функция `leader_ensure_profile(text)` больше не активирует новых пользователей автоматически.
6. Текущие активные профили не меняются.
7. Интерфейс CRM v4 показывает неактивному пользователю понятное состояние `Доступ ожидает активации` и скрывает рабочую область.
