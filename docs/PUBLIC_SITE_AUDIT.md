# Аудит публичного сайта РА «Лидер»

Дата аудита: 2026-06-26.  
Контур: только публичный сайт `https://www.lider-bsk.ru` и безопасная передача заявок в CRM через `leader-public-lead v8`.

## Проверенные источники

- GitHub: `deputat36/lider-bsk`, ветка `main`.
- Supabase: проект `ofewxuqfjhamgerwzull`.
- Edge Function: `leader-public-lead v8`, `verify_jwt=false`.
- Публичные файлы: `index.html`, `request.html`, `robots.txt`, `sitemap.xml`, `assets/public-lead-form.js`, `assets/public-lead-form.css`, основные посадочные страницы услуг.
- Документация: `README.md`, `AGENTS.md`, `docs/PROJECT.md`, `docs/STATUS.md`, `docs/DECISIONS.md`, `docs/AUTOPILOT_RULES.md`.

## Краткий вывод

Сайт уже выполняет главную задачу: объясняет услуги РА «Лидер» в Борисоглебске, ведёт пользователя к заявке и передаёт её в CRM через `leader-public-lead`. Самый сильный технический блок — форма: есть `request_id`, honeypot, защита от повторного клика, передача UTM, page URL/path/title, Яндекс Метрика и аудит в `leader_public_lead_audit`.

Главные зоны роста не в базе и не в CRM, а в публичном контуре: единый фирменный шаблон посадочных страниц, Open Graph-картинка для ВК/Telegram, показ номера обращения на всех формах, единая микроразметка LocalBusiness/Service, расширение SEO-структуры под коммерческие запросы и ручная проверка реальной заявки после v8.

## Что работает хорошо

1. **Структура сайта.** Есть главная, общие коммерческие страницы, страницы услуг, сценарные страницы для бизнесов, FAQ, цены, портфолио, политика обработки данных.
2. **SEO-база.** У основных страниц есть человекочитаемые URL, title, description и canonical. `sitemap.xml` содержит публичные страницы и не включает CRM.
3. **robots.txt.** Разрешает обход сайта и указывает sitemap.
4. **Форма заявки.** Используется общий виджет `assets/public-lead-form.js?v=4`, который отправляет заявку в `leader-public-lead`.
5. **Связь с CRM.** Edge Function пишет в `leader_leads`, добавляет аудит в `leader_public_lead_audit`, возвращает `request_id`.
6. **Идемпотентность.** В `leader_leads` есть уникальность `request_id`; v8 явно фиксирует дубли как `duplicate`.
7. **Honeypot.** Поле `website` передаётся на сервер, заполнение фиксируется в аудите как `suspicious`, а не скрывается только на клиенте.
8. **Метрика.** Яндекс Метрика подключается из общего JS, события формы отправляются через цели.
9. **Адаптивность.** Общий CSS содержит защиту от горизонтального скролла, мобильные сетки и липкую мобильную CTA-панель.

## Проблемы по приоритету

### Критичные

1. **Нужна реальная ручная проверка цепочки после v8.** В документации уже отмечено, что нужно отправить реальную тестовую заявку, записать номер обращения и проверить цепочку `сайт → leader-public-lead → leader_leads → leader_public_lead_audit → CRM`.
2. **Номер обращения показывается гарантированно только на `request.html`.** На этой странице подключён `public-lead-reference-v1.js`, который слушает `lead_sent`. На остальных формах общий виджет показывает успешную отправку, но не обязан показывать пользователю `request_id`.
3. **Open Graph неполный для ВК/Telegram.** На главной есть базовые `og:title/og:description`, но нет `og:image`. Часть посадочных страниц может не иметь полного OG-набора. Для социальных ссылок нужна единая картинка 1200×630 и мета-теги.
4. **Единообразие посадочных страниц.** Часть услуг выполнена упрощённым шаблоном с другой визуальной системой: `Arial`, сине-серый hero и жёлтая кнопка. Это отличается от фирменного чёрно-оранжевого стиля главной и формы.

### Важные

1. **Микроразметка неполная.** Есть JSON-LD на части страниц, но нужно унифицировать `LocalBusiness`, `Organization`, `Service`, `FAQPage`, `BreadcrumbList`.
2. **Privacy-страница слабее SEO-стандарта.** В `privacy.html` нет description/canonical; это не коммерческая страница, но для аккуратности лучше привести к общему минимуму.
3. **Скорость и лишний код.** Многие страницы содержат большой inline CSS. Это нормально для простого статического сайта, но усложняет поддержку и кэширование. Следующий шаг — вынести общий публичный стиль в один CSS, не меняя визуал.
4. **Cache-buster версии частично разнородны.** JS формы уже `v=4`, CSS на части страниц остаётся `v=3`, helper заявки — `v=1`. Это не ломает работу, но нужен единый журнал версий публичных ассетов.
5. **Sitemap без `lastmod`.** Для небольшого сайта не критично, но полезно добавить даты обновления после следующего контентного этапа.
6. **Контакты и доверие можно усилить.** На страницах есть телефон, но полезны блоки: адрес/география, режим связи, примеры работ, отзывы/кейсы, сроки, “что нужно для расчёта”, “как проходит заказ”.
7. **Конверсия.** Формы есть, но нужны более точные CTA под услугу: “Рассчитать баннер”, “Подобрать рекламу для магазина”, “Проверить карточку на Яндекс Картах”, “Заказать размещение в сообществах”.

### Желательные

1. **Единый публичный шаблон.** Header/footer, хлебные крошки, блок доверия, блок “что указать”, форма, FAQ.
2. **Новые посадочные страницы.** См. раздел “Контентный план”.
3. **Расширение коммерческих текстов.** Больше конкретики по материалам, размерам, срокам, типовым задачам и ошибкам клиентов.
4. **Визуальные примеры.** Добавить фото/кейсы или хотя бы карточки “до/после”, не перегружая сайт.
5. **Отдельные UTM/цели по кнопкам.** В Метрике уже есть базовые цели, но можно добавить цели по типам CTA.

## Безопасные исправления без риска для CRM и базы

1. Зафиксировать аудит и план в документации.
2. Добавить CI-проверку публичного сайта:
   - `robots.txt` и `sitemap.xml`;
   - отсутствие CRM/nav в sitemap;
   - наличие title/description/canonical на индексируемых публичных страницах;
   - подключение формы только через `public-lead-form.js?v=4`;
   - сохранение `request_id`, honeypot, UTM/page_path/submitted_at;
   - отсутствие service-role маркеров в публичных HTML/assets;
   - порядок подключения `public-lead-reference-v1.js` перед формой на `request.html`.
3. Не менять Edge Function и таблицы без отдельного этапа.
4. Не менять `nav_*`.

## План работ

### Этап 1. Безопасная фиксация и защита

- Добавить `docs/PUBLIC_SITE_AUDIT.md`.
- Обновить `docs/STATUS.md`.
- Добавить GitHub Actions проверку публичного сайта.
- Риск: минимальный, CRM и база не меняются.

### Этап 2. Форма и request_id

- Показать номер обращения после успешной отправки на всех страницах с формой, а не только на `request.html`.
- При успешном ответе читать `request_id` из ответа Edge Function, fallback — локальный `rid`.
- Не менять endpoint и payload, только текст статуса.
- Ручная проверка: отправить заявку с `request.html`, записать номер, проверить аудит и CRM.

### Этап 3. SEO и Open Graph

- Создать PNG/JPG `og-image` 1200×630 в фирменном стиле.
- Добавить `og:image`, `og:image:width`, `og:image:height`, `twitter:card`.
- Унифицировать OG title/description на всех публичных страницах.
- Добавить `lastmod` в sitemap.
- Добавить description/canonical в `privacy.html`.

### Этап 4. Микроразметка

- Главная: `LocalBusiness` / `Organization`.
- Услуги: `Service` + `BreadcrumbList`.
- FAQ: `FAQPage`.
- Цены/пакеты: `OfferCatalog` при аккуратной формулировке “от”, без обещаний фиксированных цен.

### Этап 5. Единый стиль и скорость

- Вынести общий публичный CSS из inline-стилей в `assets/public-site.css`.
- Сохранить текущий фирменный стиль: чёрный/графит/оранжевый, крупные заголовки, карточки, CTA.
- Перевести упрощённые страницы услуг на общий шаблон.
- Проверить мобильный вид.

### Этап 6. Конверсия и тексты

- Для каждой услуги добавить:
  - кому подходит;
  - что входит;
  - что нужно для расчёта;
  - сроки и ограничения;
  - частые ошибки;
  - CTA с предзаполнением формы.
- Усилить доверие:
  - телефон в hero и footer;
  - география “Борисоглебск и район”;
  - примеры работ;
  - понятный процесс заказа;
  - политика обработки данных рядом с формой.

## Что проверить вручную после безопасного этапа

1. Открыть главную и 3–5 посадочных страниц на телефоне.
2. Проверить, что кнопки “Оставить заявку” ведут к форме.
3. Открыть `request.html`, отправить реальную тестовую заявку.
4. Скопировать показанный номер обращения.
5. В CRM открыть аудит заявок, найти `request_id`.
6. Проверить виджет `Проверить request_id`.
7. Повторить техническую отправку с тем же `request_id` только отдельным тестом, чтобы убедиться в audit-событии `duplicate`.
8. Проверить в Яндекс Метрике события `form_submit_attempt`, `lead_sent`, `lead_send_error`.
