# Правила автопилота РА «Лидер»

Дата: 2026-06-27.

Этот документ фиксирует режим работы, при котором Codex максимально самостоятельно ведёт технические улучшения проекта, а владелец подключается только в спорных или рискованных местах.

Практическая инструкция для владельца: `docs/CODEX_OWNER_PLAYBOOK.md`.

## Рабочий контур

Использовать только:

- GitHub: `deputat36/lider-bsk`;
- GitHub: `deputat36/lidercalculator`, только если задача явно касается временной CRM или совместимости;
- Supabase: `ofewxuqfjhamgerwzull`;
- таблицы, функции и политики с префиксом `leader_*`.

Не использовать и не менять без отдельного решения:

- `nav_*`;
- чужие репозитории;
- старые контуры CRM, если задача явно не про миграцию или проверку совместимости.

## Режим по умолчанию

Если владелец пишет `@GitHub @Supabase продолжай автономно`, считать режим таким:

- GitHub можно менять для безопасных исправлений, документации и workflows;
- Supabase только read-only;
- production DDL/DML в Supabase запрещён без явного разрешения;
- issue #15 использовать как журнал статуса по CRM hardening;
- CI green не утверждать, если GitHub connector вернул пустые `statuses` / `workflow_runs` для push-коммита;
- после каждого автономного шага сообщать коммиты, проверки, ограничения и ручные действия владельца.

## Что можно делать самостоятельно

- исправлять явные ошибки фронтенда;
- обновлять документацию проекта;
- добавлять безопасные GitHub Actions проверки;
- усиливать диагностику CRM;
- улучшать подсказки, issue templates и чек-листы тестировщика;
- синхронизировать документацию с фактическим live-состоянием;
- читать Supabase Advisors, Edge Function metadata, grants, RLS, policies, triggers и структуру таблиц;
- создавать read-only Supabase status snapshots;
- обновлять cache-buster guards и browser-test runbooks;
- записывать результат в issue #15.

## Что требует согласования владельца

- Supabase production DDL или DML;
- удаление таблиц, данных, индексов, политик или функций;
- массовое изменение RLS;
- изменение бизнес-логики расчётов, финансов, заказов и ролей;
- деплой Edge Functions, кроме узкой согласованной hotfix-правки;
- подключение внешних платных сервисов;
- рассылки, Telegram-уведомления и любые действия от имени компании;
- реальные тестовые заявки, если они остаются в CRM;
- любые изменения в `nav_*`.

## Безопасность

- service-role ключ не должен попадать во фронтенд;
- публичная форма работает через `leader-public-lead` с `verify_jwt=false`;
- CRM-функции работают через JWT;
- обычная модель CRM-доступа: owner/admin создаёт invite во вкладке `Доступ`, пользователь без invite остаётся pending/inactive;
- любые изменения БД должны быть воспроизводимы миграцией или описаны как план, если live-DDL заблокирован;
- при блокировке инструмента нельзя имитировать изменение — нужно честно зафиксировать ограничение;
- `nav_*` предупреждения Supabase Advisor не считать исправленными работами по RA Lider.

## Формат отчёта владельцу

После автономного шага сообщать кратко:

- что изменено;
- где изменено;
- какие коммиты созданы;
- что проверено в GitHub;
- что проверено в Supabase read-only;
- что осталось непроверенным или заблокированным;
- что владельцу нужно проверить в браузере.
