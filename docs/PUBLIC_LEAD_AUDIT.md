# Аудит публичных заявок РА «Лидер»

Дата: 2026-06-23.

## Зачем нужен аудит

Аудит публичных заявок нужен для контроля связки:

`сайт → Edge Function → leader_leads → CRM`

Он помогает понять:

- была ли отправка формы;
- почему заявка не попала в CRM;
- не сработал ли honeypot;
- были ли пустые или подозрительные отправки;
- с какой страницы пришла заявка;
- какой `request_id` использовался;
- были ли ошибки записи.

## Таблица

Используется таблица:

`public.leader_public_lead_audit`

Основные поля:

- `created_at` — время события;
- `request_id` — идентификатор отправки;
- `phone_normalized` — телефон в нормализованном виде;
- `source_page_path` — путь страницы отправки;
- `page_url` — полный URL страницы;
- `user_agent` — браузер/устройство;
- `referer` — источник перехода;
- `utm_source`, `utm_medium`, `utm_campaign` — UTM-метки;
- `result` — результат обработки;
- `reason` — причина;
- `payload` — технические данные отправки.

## Результаты обработки

Возможные значения `result`:

- `accepted` — отправка принята;
- `duplicate` — дубль, если будет добавлена отдельная фиксация дублей;
- `suspicious` — подозрительная отправка, например заполнен honeypot;
- `rejected` — отклонено, например нет телефона и сообщения;
- `error` — ошибка записи заявки.

## RLS, GRANT и доступ

RLS включён.

Табличные права после ужатия 2026-06-23:

- `anon` — только `INSERT`;
- `authenticated` — только `SELECT`;
- `service_role` — служебный доступ сохранён.

Запись:

- разрешена публичному `anon`-контуру, чтобы `leader-public-lead` могла писать аудит через REST;
- insert-политика больше не использует открытый `WITH CHECK true`;
- публичная вставка ограничена ожидаемой формой события: обязательный `request_id`, допустимый `result`, JSON-object в `payload`, ограничение размера `payload`, временное окно `created_at`, лимиты длины технических полей.

Чтение:

- доступно только авторизованным активным сотрудникам из `leader_user_profiles` с ролями:
  - `owner`;
  - `admin`;
  - `manager`.

Важно:

- `anon` не должен иметь `SELECT`, `UPDATE`, `DELETE`, `TRUNCATE`, `REFERENCES`, `TRIGGER`;
- `authenticated` не должен иметь `INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, `REFERENCES`, `TRIGGER`.

## Edge Function

Функция:

`leader-public-lead`

Текущая версия:

`v6`

Что делает:

1. Принимает отправку с сайта.
2. Генерирует или принимает `request_id`.
3. Нормализует телефон.
4. Проверяет honeypot.
5. Проверяет наличие телефона или сообщения.
6. Пишет заявку в `leader_leads` с защитой от дублей по `request_id`.
7. Пишет событие в `leader_public_lead_audit`.

Важно:

- ошибка записи аудита не должна блокировать получение основной заявки;
- аудит нужен для диагностики, а не для остановки продаж.

## Как проверить

1. Открыть сайт.
2. Отправить тестовую заявку.
3. Открыть CRM v4.
4. Нажать Ctrl + F5.
5. Открыть `Аудит заявок`.
6. Нажать `Обновить аудит`.
7. Проверить, что появилась запись `Принято`.

## Что считать проблемой

Проблемы:

- заявка появилась в CRM, но нет события аудита;
- событие аудита есть, а заявки нет;
- все события имеют статус `error`;
- раздел `Аудит заявок` не появляется после Ctrl + F5;
- сотрудник с ролью `owner/admin/manager` не может прочитать аудит;
- публичная форма начала возвращать ошибку после внедрения аудита;
- `anon` имеет любые права на аудит, кроме `INSERT`;
- `authenticated` имеет любые права на аудит, кроме `SELECT`.
