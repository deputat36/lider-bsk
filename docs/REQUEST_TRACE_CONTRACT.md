# Контракт трассировки публичной заявки

Дата: 2026-06-25.

## Зачем нужен контракт

Цель — быстро проверить одну заявку от сайта до CRM без догадок.

Единая цепочка:

`сайт → leader-public-lead → leader_leads → leader_public_lead_audit → CRM`

Основной ключ связки:

`request_id`

Он должен быть одинаковым:

- в ответе сайта после отправки;
- в `leader_leads.request_id`;
- в `leader_public_lead_audit.request_id`;
- в CRM-разделе `Аудит заявок`.

## Обязательные признаки корректной цепочки

1. На сайте после отправки показан номер обращения.
2. В `leader_leads` есть строка с этим `request_id`.
3. В `leader_public_lead_audit` есть строка с этим `request_id`.
4. В CRM раздел `Аудит заявок` находит этот `request_id` через поиск.
5. Если honeypot заполнен, заявка не создаётся, но audit-событие создаётся со статусом `suspicious`.

## Что считать разрывом

### Есть заявка, но нет audit

Возможные причины:

- старая версия `leader-public-lead`;
- ошибка записи аудита;
- RLS/GRANT на `leader_public_lead_audit`;
- frontend отправляет не тот payload.

### Есть audit, но нет заявки

Возможные причины:

- honeypot;
- пустая заявка;
- ошибка вставки в `leader_leads`;
- audit зафиксировал `rejected` или `error`.

### Номер показан на сайте, но CRM ничего не находит

Возможные причины:

- браузер держит старый JS;
- поиск в аудите смотрит только последние 80 событий;
- пользователь не обновил CRM через Ctrl + F5;
- используется не тот контур CRM.

## Минимальный ручной тест

1. Открыть `https://www.lider-bsk.ru/request.html`.
2. Нажать Ctrl + F5.
3. Отправить тестовую заявку.
4. Скопировать показанный номер обращения.
5. Открыть `https://deputat36.github.io/lider-bsk/crm/v4/`.
6. Нажать Ctrl + F5.
7. Войти в CRM.
8. Открыть `Аудит заявок`.
9. Вставить номер обращения в поиск.
10. Проверить, что событие найдено.
11. Открыть `Заявки` и найти заявку по телефону/имени/услуге.
12. Сравнить `request_id` в техданных с номером обращения.

## Будущая доработка CRM

Нужно добавить отдельный раздел или виджет:

`Трассировка заявки`

Функции виджета:

- ввод `request_id`;
- поиск заявки в `leader_leads`;
- поиск события в `leader_public_lead_audit`;
- вывод статуса цепочки;
- кнопка открытия найденной заявки.

Важно: виджет должен использовать только RLS-доступ текущего пользователя и не должен требовать service-role ключей в браузере.
