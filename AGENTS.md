# AGENTS.md

## Назначение репозитория

`deputat36/lider-bsk` — основной репозиторий проекта РА «Лидер».

Здесь должны находиться:

- публичный сайт;
- CRM РА «Лидер»;
- внутренняя документация;
- миграции и описание Supabase;
- интеграции сайта и CRM.

Репозиторий `deputat36/lidercalculator` используется только для самостоятельного калькулятора. Новые модули CRM не должны разрабатываться в нём.

## Основная ветка

Основная ветка — `main`.

Все значимые изменения выполняются в отдельной ветке и передаются через pull request.

## Обязательная проверка перед задачей

Перед изменениями:

1. Проверить, что открыт `deputat36/lider-bsk`.
2. Проверить актуальность ветки `main`.
3. Прочитать `README.md`, `docs/PROJECT.md`, `docs/STATUS.md` и `docs/DECISIONS.md`.
4. Изучить связанные файлы и существующую реализацию.
5. Проверить открытые pull request.
6. Для задач с базой проверить Supabase-проект `ofewxuqfjhamgerwzull`.
7. Убедиться, что задача не реализована ранее.

## База данных

- Таблицы CRM РА «Лидер» используют префикс `leader_`.
- Таблицы `nav_*` относятся к проекту «Навигатор сделок» и не используются для данных РА «Лидер».
- Структурные изменения выполняются только миграциями.
- Запрещено удалять таблицы, поля, данные, RLS-политики, функции и триггеры без отдельного подтверждения владельца.
- Нельзя хранить service role, приватные API-ключи и пароли в GitHub.
- Публичный publishable/anon key Supabase допустим только вместе с корректным RLS.
- При изменении Supabase Auth учитывать, что он используется несколькими проектами.

## Правила разработки

- Не переписывать работающие модули без необходимости.
- Не менять публичный сайт при работе над CRM, если это не требуется задачей.
- Не менять CRM v2 во время развития CRM v4 без отдельной причины.
- Сохранять обратную совместимость действующих адресов до завершения проверки переноса.
- Не смешивать клиентские тексты с внутренними финансовыми данными.
- Себестоимость, прибыль, маржа и цены подрядчиков не должны попадать в коммерческие предложения клиенту.
- Интерфейс должен быть адаптивным для экранов шириной 360–430 пикселей.
- Основные действия должны быть понятны человеку без технической подготовки.

## Документация

После значимых изменений обновлять `docs/STATUS.md`.

При принятии архитектурного решения обновлять `docs/DECISIONS.md`.

Миграции Supabase фиксировать в статусе проекта с названием и назначением.

- Проверить изменённые файлы.
- Убедиться, что не затронуты несвязанные разделы.

- Проверить авторизацию и RLS для затронутых данных.
- Проверить отсутствие секретов.
